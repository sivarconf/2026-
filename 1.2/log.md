### K230 识别算法 v33 — 彻底移除花色补偿，改为花色总和裁决

3. **新增花色裁决**：`_resolve_suit_conflict()` 当 Top1/2 分差 < 0.20 且同颜色不同花色时，比较两花色的总置信度和
4. **JOKER_GAP_THRESH: 0.15→0.25**：更容易把 S_A 换下去

### K230 识别算法 v31 — 针对性优化7张难检牌

**问题回顾（v30 日志分析）**：
- 测试牌组：梅花10、梅花6、梅花8、方块Q、梅花9、小王、梅花K
- 确认：C_10、S_A、C_8、C_7、S_J、C_K、S_Q、JOKER_B、S_9（9张）
- **误确认：S_6（真牌是梅花6）** — Top1 始终是 S_6，C_6 从未出现
- **误确认：S_J（真牌是方块Q）** — Top1 始终是 H_J/D_Q，S_J 确认
- **丢失：D_Q（方块Q）** — D_Q 出现在 Top2/3，但从未达到 Top1
- **丢失：C_6（梅花6）** — 被 S_6 压制，从未出现

**v31 根因分析**：
1. **D_Q 丢失**：方块Q 置信度系统性偏低，SUIT_BOOST_SUITS 只有 C 和 JOKER_B，没有 D
2. **C_6 被 S_6 压制**：困难牌 Top1 与 Top2 分差极小时被降权，S_6=0.387, C_6=0.312 → gap 仅 0.075，vote_score 降权后 < MIN_CONF=0.20，无法投票
3. **S_J 被误确认**：D_Q=0.232, H_J=0.233，D_Q 应该是 Top1 但 H_J 以微弱优势获胜
4. **S_A 冷却期内误加入候选池**：S_A 确认后，冷却期内 Top1=S_A 时被加入候选池，干扰下一张牌确认

**v31 核心修复**：
1. **D_Q boost**：将 D 加入 SUIT_BOOST_SUITS `{"C", "D", "JOKER_B"}`
2. **JOKER_GAP_THRESH 降低**：0.20 → 0.15，更敏感识别 S_A↔JOKER 混淆
3. **困难牌 Top1 保护**：困难牌 Top1 时允许 Top2 不超过 Top1×1.2
4. **S_A 冷却期保护**：冷却期内 Top1 是 S_A 时不加入候选池

### K230 识别算法 v30 — JOKER_B Boost 防误触发机制

**v30 核心修复：Boost 防误触发机制**：
- `JOKER_B_MAX_GAP = 0.25`：Boost 后 JOKER_B 与 Top1 差距必须 < 0.25 才生效
- Boost 系数从 1.5 降至 1.3（温和放大，减少把普通牌误推上 Top1 的风险）

### K230 识别算法 v29 — JOKER_B Boost 机制

**v29 根因分析**：
- JOKER_B 自身 softmax 分值过低（<0.10），即使出现在帧中也过不了识别阈值
- v28 的 S_A 交换逻辑只解决"S_A 误识别为 JOKER"的问题

**v29 核心修复：JOKER_B Boost 机制**：
- JOKER_B softmax 分值 >= 0.10 时，直接乘以 1.5 倍放大
- Boost 在 S_A 交换之后执行，确保 JOKER_B 能从交换中获益

### K230 识别算法 v28 — S_A↔JOKER混淆修正 + 花色补偿优化

**v28 根因分析（小王误识别为S_A）**：
- 模型对 JOKER_B 存在系统性误识别：softmax 中 S_A 得分较高，JOKER_B 得分紧随其后
- 典型场景：S_A=0.55, JOKER_B=0.42，分差仅 0.13

**v28 核心修复：S_A ↔ JOKER_* 混淆修正**：
- 当 Top1=S_A 且 S_A-JOKER_* 分差 < JOKER_GAP_THRESH=0.20 时，交换两者 softmax 分值

### K230 识别算法 v27 — 7张难检牌专项优化

**v27 根因分析**：
- 7张难检牌中，**5张是梅花花色**，模型对梅花存在系统性置信度偏低
- Top1 与 Top2/3 的分差极小（gap < 0.08），TOP1_NOISE 大量触发
- 极低置信度帧（0.20~0.30）被硬性丢弃

**v27 三大核心优化**：
1. **花色级置信度补偿**：对梅花/小王，将同花色所有牌的 softmax 分值累加到目标牌
2. **候选池加权计数**：Top1→+1.0票，Top2→+0.5票，Top3→+0.3票
3. **TRACKING 分支动态阈值**：难检牌 vote_score >= 0.20 即投票

### K230 识别算法 v26 — 修复"一张牌检测到多张"问题

**v25 Bug 根因分析**：
- COOLDOWN 分支只过滤 `idx ∈ confirmed_cards`，不检查 rank
- 冷却期内同一物理牌的不同花色全部进入候选池 → 某花色达到2票 → 被误确认为新牌

**v26 核心修复**：
1. COOLDOWN 分支增加 rank ∈ cooldown_ranks 过滤
2. TRACKING 分支增加 rank ∈ cooldown_ranks 过滤 + 降级投票

### K230 识别算法 v25 — 难检牌专项优化

**v25 根因分析与核心修复**：
1. TRACKING 分支 Top1 被过滤时直接丢弃 → 改为降级投票
2. COOLDOWN 分支候选池积累不足 → 确保每帧至少一个候选
3. COOLDOWN 每帧检查候选池：达到阈值立即确认

### K230 识别算法 v23 — 每张牌独立 vote_count

**v23 Bug 详解**：

- D_5 和 H_3 各 1 票互不相让，H_3 永远无法确认

**v23 核心修复**：
- 每张牌有独立的 `vote_counts[idx]` 计数器
- 每帧只有 Top1 计入对应牌的计数器

### K230 识别算法 v22 — 候选池机制

**v22 Bug 详解**：
- 冷却期内 cls_idx=-1 的帧不加入 vote_window，导致冷却期结束后窗口为空

**v22 核心修复：引入候选池机制**：

- 冷却期结束分析候选池，达到阈值则立即确认
