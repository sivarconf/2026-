# 代码修改日志

## 2026-05-18

### 冷却期结束后重复识别同一张牌 Bug 修复

**问题**：`StabilityStateMachine` 的 COOLDOWN 分支中，冷却期结束后会清空 `confirmed_cards` 和 `confirmed_ranks`，导致同一张牌在冷却期结束后可以被再次确认。

**修复**：冷却期结束后只重置追踪状态（`vote_window`、`vote_counts`等），**保留** `confirmed_cards` 和 `confirmed_ranks`，使排除记录在整次运行中永久生效。

### 滑动窗口投票机制实现

**修改内容**：`StabilityStateMachine` 从连续帧确认改为滑动窗口投票：

1. **参数变更**：`CONSECUTIVE_THRESH=3` → `WINDOW_SIZE=5` + `VOTE_THRESH=3`
2. **数据结构**：用 `vote_window` 列表（最近5帧的 Top1 idx）+ `vote_counts` 字典（各牌出现次数）替代 `current_candidate_consec`
3. **投票逻辑**：每帧先将过滤后的 Top1 入队，窗口满时弹出旧帧；检查是否有牌出现次数达到3次即确认
4. **OSD 显示**：适配新数据结构，显示投票进度条和窗口内容、置信度总和

**修改文件**：`K230/main.py`

### 投票平票时以置信度优先

**修改内容**：`StabilityStateMachine` 新增 `vote_scores` 字典记录窗口内各牌的置信度总和。

**平票裁决逻辑**：
- 多张牌同时达到3票时，首先选**票数最多**的牌
- 若票数相同，则选**置信度总和最高**的牌
- 优先确保识别结果更可靠

**修改文件**：`K230/main.py`

### CARD_COVER_THRESH 死代码修复

**问题**：`app_line.c` 第 154 行 `CARD_COVER_THRESH = LOST_THRESH_SHORT`（均为 30），导致判断链第二个 `else if (s_lost_cnt < CARD_COVER_THRESH)` 永远无法进入，扑克牌遮线专用逻辑（150ms~300ms 圆弧保持）是死代码。小车在 150ms 后直接进入中等丢线降速模式，而非圆弧保持，在圆形赛道上造成偏轨。

**修复**：
1. `stm32/Core/Src/app_line.c` — `CARD_COVER_THRESH` 从 30 改为 60（300ms）
2. `竞赛小车工程总说明书.md` §5.2.4 丢线保护策略表格更新：明确 150ms~300ms 为圆弧保持，300ms~500ms 为中等丢线降速

**修复后判断链**：
- `< 30` (150ms) → SHORT，保持上一 error 直行
- `< 60` (300ms) → CARD_COVER，保持遮线前角度沿圆弧行进（生效！）
- `< 100` (500ms) → MED，降速找线
- `> 600ms` → ERROR 停车

## 2026-05-17

### B题第二圈停车后声光无法自动停止

**问题**：B题跑完第二圈后，蜂鸣器和 LED 持续响/亮，无法自动关闭。

**根因**：`s_alert_off_done` 标志在第一圈声光提示 1 秒后被置 1，但 `App_Line_Init()` 在再次运行时未重置该标志。`App_Line_CheckAlertOff()` 的条件是 `s_alert_done && !s_alert_off_done`，由于 `s_alert_off_done` 已是 1，条件恒为假，导致第二圈停车时重新打开的蜂鸣器和 LED 永远无法被关闭。

**修复**：
1. `app_line.c` - 第二圈/一圈完成触发声光时，主动重置 `s_alert_off_done = 0`（双重保障）
2. `user_main.c` - K4 手动停车时，立即关闭蜂鸣器和 LED

**修改文件**：`stm32/Core/Src/app_line.c`、`stm32/Core/Src/user_main.c`

### K230 识别算法 v31 — 针对性优化7张难检牌

**问题回顾（v30 日志分析）**：
- 测试牌组：梅花10、梅花6、梅花8、方块Q、梅花9、小王、梅花K
- 确认：C_10、S_A、C_8、C_7、S_J、C_K、S_Q、JOKER_B、S_9（9张）
- **误确认：S_6（真牌是梅花6）** — Top1 始终是 S_6，C_6 从未出现
- **误确认：S_J（真牌是方块Q）** — Top1 始终是 H_J/D_Q，S_J 确认
- **丢失：D_Q（方块Q）** — D_Q 出现在 Top2/3，但从未达到 Top1
- **丢失：C_6（梅花6）** — 被 S_6 压制，从未出现

**v31 根因分析**：
1. **D_Q 丢失**：方块Q 置信度系统性偏低，SUIT_BOOST_SUITS 只有 C 和 JOKER_B，没有 D
2. **C_6 被 S_6 压制**：困难牌 Top1 与 Top2 分差极小时被降权，S_6=0.387, C_6=0.312 → gap 仅 0.075，vote_score 降权后 < MIN_CONF=0.20，无法投票

**v31 核心修复**：
1. **D_Q boost**：将 D 加入 SUIT_BOOST_SUITS `{"C", "D", "JOKER_B"}`
2. **JOKER_GAP_THRESH 降低**：0.20 → 0.15，更敏感识别 S_A↔JOKER 混淆

4. **S_A 冷却期保护**：冷却期内 Top1 是 S_A 时不加入候选池

**v29 根因分析**：

### 2026-05-20 | main_new2.py VOTING 窗口判断 bug 修复

**问题**：`VOTING` 状态下窗口结束条件用 `voting_frames_collected >= WINDOW_SIZE`，但该计数器只在低置信度帧（cls_idx < 0 或无 valid candidates）时不递增，导致低置信度帧出现时窗口无限拉长。

**修复**：将窗口结束条件改为 `round_total_frames >= WINDOW_SIZE`（每帧必递增），并修正 `_tick_finalize` 调用以避免 `cls_idx < 0` 时引用未定义变量 `valid`。

### 2026-05-20 | main_new2.py DISCARDING 帧数超时修复

**问题**：`DISCARDING` 状态下 `cls_idx < 0` 或无 valid candidates 时直接 return，`round_total_frames` 未递增，帧配额被浪费，导致无限等待。

**修复**：入口统一递增 `round_total_frames`，设置总超时上限 `DISCARD_FRAMES + WINDOW_SIZE = 9` 帧，超时后强制进入 VOTING（重置投票状态后继续，不丢弃本轮）。

### 2026-05-21 | app_line.c F题参数统一

**修改**：挑战题（F题）不再区分稳定模式和高速模式，统一使用一套参数：PWM=600，KP=6.0，KI=0，KD=4.0。基础题（B题）两种速度及其他参数保持不变。

### 2026-05-21 | main_new2.py v22 回顾一致性检查（S_10 vs C_8 覆写）

**目的**：解决 S_10 和 C_8 容易互相误识别的问题。当投票确认 S_10 后，对本轮投票历史做回顾检查，三条件全满足则覆写为 C_8。

**条件**：
- 条件 A：S_10 当 top1 的比例 < 85%（有其他类抢过 #1）
- 条件 B：C_8 分数 > 0.05 出现在 Top3 中出现过 >= 3 帧（C_8 不是偶然露脸）
- 条件 C：S_10 > 0.7 的高置信帧 < 3 帧（S_10 缺乏持续高分）

**修改点**：
- `StabilityStateMachine.__init__`：增加 `C_8_idx` 属性
- 新增 `_retrospective_s10_vs_c8_check(confirmed_idx, history)` 方法
- `_tick_confirm` / `_tick_finalize`：在确认时调用回顾检查，追加 `should_override` 返回值
- `tick` 方法：所有返回语句增加第 8 元素 `should_override=False`
- `vote_history`：每帧记录增加 `"top3"` 字段用于条件 B 检测
- 主循环：确认后检查 `should_override`，为 true 时将结果覆写为 C_8
- 版本升至 v22 / 2.2
